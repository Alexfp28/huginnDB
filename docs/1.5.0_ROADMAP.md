# 1.5.0 roadmap

Triage round sourced from four issues filed the same day (#11-#14) plus one bug
reported by email (now #15). Unlike 1.4.0, none of this is pre-planned
roadmap work — it's reactive fixes/small features surfaced by real usage
after the workspaces→windows migration. This document records the
root-cause investigation for each item (file:line references) so
implementation can start directly without re-discovering the mechanism.

See `CLAUDE.md` for architecture and gotchas #8, #15, #19 (directly relevant
here), and `CHANGELOG.md` for the 1.4.0 entry.

## Planned

### 1. Scope per-window event broadcasting (#11, and the display-layer half of #13) — done

Implemented: `log_bus::emit` now takes the originating window's label and
uses `AppHandle::emit_to` instead of a broadcast `emit`. Every command that
produces a SQL or connection-lifecycle log entry gained a `tauri::Window`
parameter (Tauri auto-injects it from the invoking webview — zero frontend
change needed) to supply that label: `execute_query`, `execute_batch`,
`fetch_table_data`, `update_cell`, `delete_rows`, `insert_row` in
`commands/query.rs`, and `connect`, `disconnect`, `test_connection`,
`open_database_view` in `commands/connection.rs`. Keepalive's own
ping-failure log line stays a broadcast (new `log_bus::broadcast`) since it
has no originating window — it reports on a connection any window may be
browsing — and its separate `connection-lost` event (the reconnect UX) was
already a correct broadcast and needed no change. See `CHANGELOG.md`
Unreleased.

**Confirmed root cause.** Every backend-to-frontend push in this app is a
process-wide broadcast, and every window — main or any "New window" secondary
— mounts the identical `App.tsx` and independently subscribes to the same
events, with no window-label guard anywhere in this path:

- `src-tauri/src/log_bus.rs:133` — `app.emit(LOG_EVENT, entry)` is plain
  `AppHandle::emit`, delivered to *every* open webview, not just the one
  whose command produced the log line.
- `src/lib/log-bridge.ts:16-18` + `src/App.tsx:480-491` — `startLogBridge()`
  is mounted unconditionally in a bare `useEffect(..., [])`, unlike
  `src/stores/persistedTabs.ts:122`'s
  `if (getCurrentWindow().label !== "main") return;` guard (gotcha #8's
  pattern — this path never got the same treatment).
- `src/stores/logs.ts` (`useLogs`) is ordinary unscoped Zustand state — each
  window's store instance is a genuinely separate JS runtime, but both
  receive and append the same broadcast entries. Net effect: a query run in
  the main window shows up in a secondary window's Console too, reading as
  "the secondary console is a pointless copy of the main one" (#11's title).
- `src-tauri/src/keepalive.rs:85` emits `CONNECTION_LOST_EVENT` via the exact
  same unscoped `app.emit` pattern, and `src/lib/connection-health-bridge.ts`
  is mounted in `App.tsx:497-508` with the same missing guard. Grepping
  `src-tauri/src` for `.emit(` turns up only these two call sites plus
  `lib.rs:289` — none use `emit_to`/`WebviewWindow::emit`.

**Relationship to #13:** the maintainer's hypothesis was "fixing #11 fixes
#13." That's confirmed at the *mechanism* level — both bugs are the same
unscoped-broadcast pattern — but the direction doesn't fully match a strict
"keepalive only works in the main window" symptom: broadcast predicts
*over*-delivery to all windows, not under-delivery to secondary ones. Also,
`connect` (`src-tauri/src/commands/connection.rs:254`) is a single Tauri
command with no window parameter — `crate::keepalive::spawn(...)` at line 330
runs identically regardless of which window's IPC call reached it, since
`AppState`/`ActiveConnections` is process-global. So the keepalive heartbeat
itself is not window-scoped and doesn't "only work in the main window" at
the backend level.

Best read: the confusion the maintainer traced to #13 is downstream of the
same broadcast leakage — a secondary window showing stale/duplicated
log-and-status noise reads as "keepalive isn't reporting correctly here."
Fixing the broadcast scoping should resolve the *perceived* #13 symptom.
If a genuine case of "heartbeat fails to detect a truly dead connection"
still reproduces after this fix, that would be a distinct, currently
unconfirmed bug — reopen rather than assume it's covered.

- **Fix shape:** give `LogEntry` (and the connection-lost payload) an
  originating-window label, threaded through every Tauri command that calls
  `log_bus::emit`/the keepalive emit (commands get a `tauri::Window` or
  `AppHandle` + label already available at the call site — no new plumbing
  needed to know the label, just to pass it through). Switch the emit calls
  to `WebviewWindow::emit` (or filter server-side) targeting only that
  window, OR — simpler, no backend change — keep the broadcast but have each
  frontend bridge ignore events not tagged for its own
  `getCurrentWindow().label`. Prefer the frontend filter: it's a one-line
  guard per bridge (mirroring `persistedTabs.ts:122`) instead of rewiring
  three emit call sites and every command that triggers a log line.
- **Blast radius:** `src/lib/log-bridge.ts`, `src/lib/connection-health-bridge.ts`,
  possibly `src-tauri/src/log_bus.rs` if the label has to originate
  server-side (a log line produced by a background task like keepalive has
  no "requesting window" — those are inherently global and should stay
  broadcast; only *command-triggered* entries have a real originating
  window). Decide per-entry-kind whether scoping even applies before
  implementing.
- **Complexity:** small-medium.

### 2. Windows taskbar icon disappearing (#12)

**Investigated, no in-repo defect found.** `tauri.conf.json` bundle icon
config, `build.rs` (plain `tauri_build::build()`, no custom resource
handling), and `icons/icon.ico` (parsed directly: 6 resolutions — 16/24/32/
48/64/256 — a reasonable set, missing only the rarely-needed 20/96) all look
normal. Zero `set_icon` calls and no tray icon anywhere in
`src-tauri/src` — nothing in this codebase mutates the icon at runtime, so
there's no code path capable of clearing it.

**Assessment: environmental**, most likely Windows Explorer's icon-cache
database corrupting/evicting entries over long uptimes (a known OS-shell bug
class affecting many unrelated Win32/Electron/Tauri apps), possibly
compounded by WebView2's multi-process model across sleep/wake cycles.

- **Action for 1.5:** no code fix planned. Close #12 with the explanation
  above, or downgrade to "won't fix — platform issue" with the standard
  mitigation (rebuild the Windows icon cache / restart `explorer.exe`).
  Optional, unconfirmed speculative hardening if the maintainer wants belt-
  and-suspenders anyway: an explicit `.icon(...)` call in `open_new_window`
  (`src-tauri/src/commands/connection.rs:894-908`) to force re-assertion of
  the icon handle on secondary windows — not a diagnosed fix, skip unless
  the symptom recurs after closing this out.
- **Complexity:** none (or trivial if the speculative hardening is wanted).

### 3. Group/folder connections (#14) — done

Implemented as scoped with the maintainer: single group per connection
(`ConnectionProfile.group: Option<String>`, free text, no separate group
registry). **Important correction made mid-implementation:** the original
plan targeted `src/components/ConnectionList.tsx` as "the sidebar", but
that component is dead code — grep confirms it's never imported anywhere in
the current tree (`App.tsx` moved connection switching to the status-bar
dropdown, `StatusConnections.tsx`, when the sidebar was redesigned into the
Schema/Saved/Workspace/Console dockview; nobody deleted the orphaned file).
Grouping was implemented in `StatusConnections.tsx` instead — the dropdown
now buckets its Active and Available sections by group with collapsible
headers (collapse state shared across both sections per group name, and
persisted in `prefs.json`). `ConnectionList.tsx` itself was left untouched;
it's a candidate for deletion in a separate cleanup pass, not bundled into
this feature. See `CHANGELOG.md` Unreleased.

Original scoping (for reference):

**No existing data model for this.** `ConnectionProfile`
(`src-tauri/src/state.rs:42-88`; mirrored in `src/types.ts:53-78`) has no
group/folder/tag field today: `id, name, driver, host, port, database,
username, ssl, ssh_tunnel?, connection_string?, auth_source?, ephemeral?`.
`src-tauri/src/store.rs` persists `Vec<ConnectionProfile>` to
`profiles.json` with no version/migration scaffold (unlike `tab_state.rs`'s
v1/v2/v3) — every field added historically is additive via
`#[serde(default)]`, so a new optional field is a non-breaking change.
`src/components/ConnectionList.tsx:134` renders `profiles` as one flat,
unsorted `.map` — no grouping/folder/favorite concept exists in the UI
either. Distinct from the removed 1.4.0 "workspaces" concept (that was
tab-layout state, not connection organization).

- **Fix shape:**
  - Backend: add `group: Option<String>` to `ConnectionProfile`
    (`#[serde(default)]`, no migration needed). Per gotcha #14, mirror it in
    `src/types.ts` in the same change — a frontend-only field silently drops
    on the next Rust struct rebuild.
  - Commands: none need signature changes — `list_profiles`, `save_profile`,
    `delete_profile`, import/export (`src-tauri/src/commands/connection.rs:82,
    97, 149, 574, 618, 715`) all pass the struct by value.
  - Frontend: bucket by group with `useMemo` in `ConnectionList.tsx` (gotcha
    #1 — don't add a derived array to the Zustand store itself), collapsible
    group headers, a group-assignment field in `ConnectionDialog.tsx`.
    Collapse/expand state is UI preference, not data — persist it in
    `prefs.json` alongside other grid/editor prefs, not in `profiles.json`.
- **Open questions to resolve before implementing** (ask the maintainer):
  single group per connection or multiple tags? Free-text group names or a
  small managed registry? Default group for ungrouped connections, or just
  an "Ungrouped" bucket?
- **Complexity:** medium (schema is trivial; the UI — drag-to-group or a
  picker, collapse state, empty-group handling — is the real surface area).

### 4. MySQL BIT insert failure (#15, reported by email) — done

**Not reproducible on current HEAD.** `insert_row`
(`src-tauri/src/commands/query.rs:1408-1562`) already mirrors
`update_cell`'s BIT handling exactly: a `RowValue.column_type` field, the
same `starts_with("BIT")` check, `CAST(? AS UNSIGNED)` placeholder
construction, and reuse of `normalize_bit_value`
(`query.rs:1219-1225`). On the frontend, `BitInput.tsx` is a dedicated
`<select>` (used identically by the insert draft row and inline edit) that
structurally can only emit `"0"`, `"1"`, or `null` — there's no path that
sends the literal string `"true"`. `TableDataTab.tsx`'s `onDraftCommit`
does propagate `columnType` for every inserted value.

This protection shipped in commits `7bca37e`/`7f88797` (2026-06-11), present
in every tagged release from **v1.1.0** onward.

- **Most likely explanation:** the report is from a pre-1.1.0 build, where
  BIT columns used the plain text `CellInput` with no `CAST`/normalization —
  typing `true` there sends a literal 4-byte string through an uncast bound
  param, which is exactly what throws "Data too long for column" on a
  `BIT(1)`.
- **Secondary, still-live possibility:** a metadata-availability edge case —
  if catalog metadata (`cols`) is empty/stale *and* there's no prior query
  result for that table, `columnType` can resolve to `undefined` even on
  current HEAD, and the backend then sends the value uncast.
- **Action for 1.5:** confirm the reporter's exact version and whether their
  insert-row `enabled` cell renders as a dropdown or free text (instantly
  distinguishes pre-fix from the metadata-gap edge case). If the metadata
  gap is confirmed, harden `insert_row` to fall back to a catalog lookup (or
  reject with a clear error) when `column_type` is missing for a BIT column,
  instead of silently sending an uncast value.
- **Complexity:** small (version confirmation), or small-medium if the
  metadata-gap hardening turns out to be needed.

**Resolved.** The maintainer confirmed the insert-row `enabled` cell *was* a
dropdown, on **1.4.0** — ruling out the pre-1.1.0 explanation and confirming
the metadata-gap edge case as live on current HEAD. Implemented the hardening:
both `insert_row` and `update_cell` now fall back to a `list_columns_inner`
catalog lookup when the frontend's `column_type` hint is `None`, instead of
trusting it blindly. `insert_row` only pays for the extra round-trip when at
least one value actually lacks a hint. See `CHANGELOG.md` Unreleased.

## Second triage round (#16-#18, filed by external users)

Filed after the first round shipped. Same reactive-fix pattern; investigated
and implemented in one pass since the maintainer approved all three,
including the full #18 event bridge rather than the minimal idempotency
patch alone.

### 5. Create database (#16) — done

Confirmed full feature gap (no backend command, no UI entry point anywhere).
Added `create_database` (`commands/schema.rs`, Postgres/MySQL only,
`validate_ident`-guarded) and a "+" button opening a `CreateDatabaseDialog`
(mirrors the existing `RenameTableDialog` pattern). Initially scoped to
multi-DB mode only, reasoning a single-DB profile has no local list to show
the new database in — the maintainer pushed back on that as needlessly
limiting (a single-database profile is at least as common as multi-DB
browsing), so the button was added to `SingleDbExplorer`'s root header too;
on success it shows a confirmation message (`createdSingleDb`) instead of a
list refresh, since there's nothing visible to update. SQLite/MongoDB get no
button anywhere (no `CREATE DATABASE` concept for either). See
`CHANGELOG.md` Unreleased.

### 6. Resizable grid columns (#17) — done

Confirmed zero resize support — not even a fixed width, just browser
auto-layout, with a `max-w-md` cap on the cell content `<div>` that would
have fought any width change anyway. Wired up TanStack Table's resize API
(`enableColumnResizing`, `columnResizeMode: "onEnd"`, drag handles,
`table-fixed` layout so explicit `<th>`/`<td>` widths actually take effect).
Persisted per-table in a new `prefs.json` field (`grid.columnWidths`,
`HashMap<"schema.table", HashMap<column, px>>` — tiny entries, no pruning
needed unlike `tab_state.json`'s query bodies). Ad-hoc query grids (no
stable table identity) resize in-session only. See `CHANGELOG.md`
Unreleased.

### 7. Multi-window state sync (#18) — done, full bridge

Both sub-complaints traced to the same root cause: every window shares one
backend `AppState`, but nothing ever pushed a change made in one window's
frontend out to the others. #18a additionally had a confirmed backend
hazard beyond that: `ActiveConnections::insert`'s replace-on-conflict
semantics meant a second window connecting an already-active profile tore
down the first window's pool and SSH tunnel. #18b's severity was data loss,
not just staleness — each window's debounced preferences save sends the
*entire* blob, so two windows editing different settings would silently
overwrite whichever saved first.

Implemented the full bridge rather than the minimal idempotency-only patch:
- `connect` no-ops (reuses the existing pool) when `ActiveConnections::contains`
  is already true, instead of falling through to `insert`'s replace path.
- `connect`/`disconnect`/`save_profile`/`delete_profile`/`import_profiles`
  broadcast `connection-opened`/`connection-closed`/`profiles-changed`;
  `update_preferences` broadcasts `prefs-changed` with the full persisted
  snapshot. All unscoped (`app.emit`, not `emit_to`) — deliberately the
  opposite choice from the #11 fix, since these describe genuinely global
  state every window should agree on, not per-window activity.
- New frontend bridges `connection-sync-bridge.ts` / `prefs-sync-bridge.ts`
  (same one-shot-listener-mounted-in-App.tsx pattern as the existing
  `connection-health-bridge.ts`).
- `stores/connections.ts`'s `connect()`/`disconnect()` had their local
  side-effects factored out into `markConnected`/`markDisconnected` so the
  sync-bridge path and the local path run identical cleanup — including the
  multi-DB synthetic-child (`<id>::db::<db>`) tab/schema sweep, which matters
  just as much when the pool died via another window's disconnect.
- `stores/preferences.ts` gained `applyExternal`, which adopts a
  broadcasted snapshot without calling `scheduleSave` — the payload already
  hit disk, so re-saving would be redundant at best and a race against a
  newer local edit at worst.

See `CHANGELOG.md` Unreleased.
