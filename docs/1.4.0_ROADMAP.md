# 1.4.0 roadmap

HuginnDB 1.3.0 shipped a triage round of fixes and small features (FK "go to
referenced row", "New query here" on a database, the issue-reporter browser fix,
and ad-hoc BIT/integer DML over the unprepared protocol). **1.4.0** was
deliberately split off as the larger, riskier cycle so the 1.3.0 history stays
legible. This document records its two planned work items — the intent, the
blast radius, and the technical hooks each depends on — so the next session can
pick them up without rediscovering the context.

See `CLAUDE.md` for the architecture and `CHANGELOG.md` for the 1.3.0 entry.

## Planned

### 1. Remove workspaces → native windows — done

Implemented as described below, with the frontend-side scoping decisions
confirmed in the same session: "New window" opens **blank** (no connection);
secondary windows are **ephemeral** (nothing about them persists across an
app restart — only the main window's tab state does); the CLI second-launch
dialog still asks "this window vs new window" but gained a "don't ask
again" toggle (`prefs.ui.cliConnectDefault`). See `CLAUDE.md` gotcha #8.

Two bugs surfaced during manual testing and were fixed in the same cycle:
`open_new_window` deadlocked/rendered blank on Windows until it was made an
`async fn` (a documented WebView2 issue — sync commands that build a
`WebviewWindow` hang, see
[tauri#13963](https://github.com/tauri-apps/tauri/issues/13963)); and a CLI
ad-hoc launch without `--password` silently created a disconnected profile
instead of attempting the connect. The top bar's File/View menus were also
split into four (File/Window/Help added) since New window and the
second-launch plumbing gave File yet another unrelated concern.

Replace the whole "workspaces" concept with real, separate OS windows
("ventanas"). The current model is broken: opening a second instance prompts
"new workspace vs current" and that flow doesn't work, and workspaces were only
ever a stand-in for the per-window instances originally intended.

- **Migration decision (confirmed with the maintainer):** the main window
  inherits the connections/tabs of the **active** workspace; the other
  workspaces are discarded. Flat per-window model from there on. Add a v2→v3
  migration in `tab_state.rs` that flattens the active workspace and drops the
  rest — keep the migration-with-test pattern used for v1→v2.
- **Blast radius (from this session's exploration):**
  - Frontend to delete/rewrite: `src/stores/workspaces.ts`,
    `src/components/WorkspaceSwitcher.tsx`, and
    `src/components/CliConnectChoiceDialog.tsx` (the "new workspace vs current"
    dialog that doesn't work).
  - Persistence scope: `src/stores/persistedTabs.ts` and
    `src-tauri/src/tab_state.rs` move from scoping by `active_workspace()` to
    scoping **per window**. `src-tauri/src/commands/prefs.rs`
    (`get_tab_state` / `save_tab_state` / `clear_tab_state`) and
    `src-tauri/src/commands/workspaces.rs` get reworked or removed.
  - Real multi-window: `src-tauri/tauri.conf.json` (one window today) plus the
    single-instance handler in `src-tauri/src/lib.rs` (which currently
    *consolidates* a second launch) → spawn a new `WebviewWindow` on demand.
    `AppState` stays global (connection pools are keyed by `connectionId` and are
    correctly shared); only the tab state needs partitioning per window.
  - UI: replace the topbar `WorkspaceSwitcher` with window management
    (e.g. "New window").
- **Why it's the big item:** it touches the frontend state tree *and* the
  backend scoping model, and it's a UX-breaking change — hence its own cycle.

### 2. Server-side users / permissions introspection — implemented

Backend, types, and UI are done for **every** driver (not just Postgres/MySQL
as originally scoped below) — see the `CHANGELOG.md` Unreleased entry for the
per-driver detail (Postgres, MySQL, MongoDB, and SQLite's explicit
no-user-model empty state). Only the release close-out (version bump,
`CHANGELOG.md` date) is still pending — deliberately held back per the
maintainer until the cycle is ready to ship.

Let HuginnDB read user-level configuration **from the database server** — list
the users/roles the connection can see and inspect their privileges. This is the
reframing of the original "user-level configuration" request.

- **Pattern to mirror:** the existing `databases → tables → columns → indexes`
  introspection, dispatched with a `match DbPool` per driver.
- **Hooks:**
  - Backend: add `list_users` + `list_privileges` to
    `src-tauri/src/commands/schema.rs`, registered in `src-tauri/src/lib.rs`
    (`generate_handler!`). PostgreSQL: `pg_roles` /
    `information_schema.role_table_grants`. MySQL: `mysql.user` / `SHOW GRANTS`.
    SQLite: empty (no user concept). MongoDB: `system.users` if in scope.
  - Types: `UserInfo` / `PrivilegeInfo` in Rust + `src/types.ts`; typed wrapper
    in `src/lib/tauri.ts`.
  - UI: a "Security" panel in the explorer (TanStack Table), lazy-loading a
    user's privileges on expand.

### 3. Connection keepalive + lost-connection reconnect — implemented

Ad-hoc addition surfaced by user reports of connections dropping from
inactivity, not part of the original two-item scope above. Investigated
first: HuginnDB had **no** proactive keepalive — no `idle_timeout`, no
heartbeat, nothing — relying entirely on `sqlx`'s default
`test_before_acquire`, which only validates a connection the next time it's
*used*, and does nothing for an idle pool sitting between user actions, or
for a dropped SSH tunnel.

- **Backend** (`src-tauri/src/keepalive.rs`, new module): every top-level
  profile connection (`connect` in `commands/connection.rs`) gets a
  background task pinging its pool every 3 minutes (`SELECT 1` / driver
  equivalent) — short enough to stay under typical NAT/load-balancer idle
  timeouts. A failed ping emits `huginndb://connection-lost` once and ends
  the loop; the dead pool is left in place until the user reconnects.
  Lifecycle mirrors `SshTunnelHandle`: a `KeepaliveHandle` cancellation
  token stashed on `ActivePool`, dropped automatically when the pool is
  removed or replaced — no separate bookkeeping in `AppState`. Deliberately
  scoped to top-level connections only; the synthetic per-database pools
  from `open_database_view` (multi-DB browsing) share their parent's
  liveness and are cheap to reopen, so a second heartbeat there would be
  redundant.
- **Frontend**: `stores/connectionHealth.ts` + `lib/connection-health-bridge.ts`
  wire the event in (mounted once in `App.tsx`, same pattern as the log
  bridge). The connection list (`ConnectionList.tsx`) and the status-bar
  connections dropdown (`StatusConnections.tsx`) both turn the status dot
  red and swap the normal connect/disconnect button for a "reconnect" one
  when a connection is flagged lost. Reconnecting calls the store's raw
  `disconnect` + `connect` (not the UI wrapper that also closes tabs), so
  open tabs and schema-tree expansion survive — the same connection id
  just gets a fresh pool.
- Chosen over a fully automatic/silent reconnect or a blocking modal, per
  the maintainer: proactive keepalive to prevent most drops, reactive
  detection via the same heartbeat for the rest, manual one-click
  reconnect rather than auto-retry.
